## Контекст задачи (Task context)

### Запрос и ожидаемый результат [M]

- Кратко по запросу: реализовать в текущем проекте расширяемую систему локализации для Telegram-бота, соблюдая SOLID, KISS, DRY, GRASP, high cohesion, low coupling.
- Ожидаемый результат:
  - Тексты интерфейса и сообщений бота берутся через единый слой локализации.
  - Переводы подгружаются динамически из JSON и отдельного metadata-файла для каждой локали.
  - Добавление новой локали выполняется добавлением файлов локали без изменений кода ядра локализации.
  - Начальные локали в проекте: русский (`ru`) и английский (`en`).

### Границы области работ [M]

- В scope:
  - Проектирование и внедрение модульного слоя локализации (контракты, сервис, загрузчик ресурсов).
  - Поддержка динамического обнаружения и загрузки локалей из файловой структуры проекта.
  - Введение metadata-схемы локали и валидации структуры переводов.
  - Интеграция выбора/резолва языка пользователя с fallback-логикой.
  - Перенос пользовательских текстов бота на ключи локализации для RU/EN.
  - Полная миграция всех hardcoded пользовательских строк проекта на централизованную систему локализации.
  - Тесты и документация по добавлению новых локалей без правки ядра.
- Вне scope [O]:
  - Полная локализация всех внутренних developer-документов проекта.
  - Изменение бизнес-логики генерации/скачивания, не связанной с текстами UI.
  - Поддержка плюрализации/ICU-форматов сверх текущих потребностей, если это не требуется существующим UI.

### Основные цели и ограничения [M]

- Файлы/компоненты:
  - `app_context.py`, `handlers/registry.py`, `core/runtime.py`, `core/runtime_snapshot.py`, `core/user_preferences.py`
  - Пользовательские UI/handler-модули: `core/ui_copy.py`, `core/ui_kit/*`, `handlers/common*.py`, `handlers/presets*.py`, `handlers/download*.py`, `handlers/prompt_editor*.py`
  - Новые модули локализации в слоях `domain/`, `application/`, `infrastructure/` (с учетом текущей layered migration)
  - Ресурсы локалей: новая директория `locales/` (или согласованный путь в проекте) с RU/EN файлами
- Системы/области: backend bot runtime, presentation layer (Telegram UI), конфигурация ресурсов, тесты, документация.
- Ограничения и допущения [O]:
  - Базовая локаль по умолчанию: `ru` (как текущий язык большинства существующих сообщений).
  - При отсутствии ключа в активной локали выполняется fallback на базовую локаль.
  - Callback data и сценарные переходы не меняются, чтобы не ломать FSM/навигацию.

## Поэтапный план (Stage-by-stage plan)

### Этап 1 - Спецификация локализации и контрактов

#### Цель этапа [M]

- Зафиксировать архитектурный контракт локализации, модель ключей и формат metadata/JSON для локалей.

#### Подзадачи [M]

- [ ] Провести инвентаризацию пользовательских текстов в ключевых `core/*` и `handlers/*` модулях.
- [ ] Сформировать схему ключей переводов (namespace + правила именования) без дублирования (DRY).
- [ ] Описать обязательные поля metadata-файла локали (код языка, display name, native name, версия/совместимость, default flag при необходимости).
- [ ] Определить доменные контракты локализации (интерфейсы/протоколы для resolver/service/loader) с низкой связанностью между слоями.

#### Почему этап не ломает общую работоспособность проекта [M]

- Этап проектный и организационный: вводит контракты/структуру, не меняя критическую runtime-логику сценариев бота.

#### Проверки/валидация этапа [M]

1. Проверить, что покрыты все категории пользовательских текстов: статические, параметризованные, alert/reply/keyboard labels.
2. Проверить, что контракт позволяет добавлять новую локаль только через ресурсные файлы.

#### Ожидаемые артефакты [O]

- Черновой key map и структура локалей (в рамках кода/документации задачи).

### Этап 2 - Ядро локализации и динамический загрузчик ресурсов

#### Цель этапа [M]

- Реализовать расширяемое ядро локализации и загрузку локалей из JSON + metadata без hardcode языков в бизнес-коде.

#### Подзадачи [M]

- [ ] Добавить доменные модели/контракты локализации (например, `LocaleMetadata`, `TranslationBundle`, `TranslationProvider`).
- [ ] Реализовать инфраструктурный загрузчик локалей из директории ресурсов с валидацией metadata и JSON структуры.
- [ ] Реализовать application-сервис локализации (`t(key, locale, params)` + fallback + безопасная подстановка параметров).
- [ ] Интегрировать сервис в сборку зависимостей (`app_context`) и передать его в registry/deps без циклических связей.

#### Почему этап не ломает общую работоспособность проекта [M]

- Изменения добавочные: новый слой локализации внедряется с fallback по умолчанию, что исключает падение UI при неполной миграции.

#### Проверки/валидация этапа [M]

1. Юнит-тесты загрузчика: корректные локали, отсутствующий/битый metadata, отсутствующий/битый JSON.
2. Юнит-тесты сервиса: fallback, форматирование параметров, поведение при отсутствии ключа.
3. Регрессионный запуск существующих тестов на отсутствие поломок.

#### Ожидаемые артефакты [O]

- Новые модули локализации в `domain/`, `application/`, `infrastructure/`.

### Этап 3 - Резолв языка пользователя и переключение локали

#### Цель этапа [M]

- Добавить устойчивый механизм выбора языка пользователя (persisted preference + fallback) и точку переключения локали в UI.

#### Подзадачи [M]

- [ ] Расширить модель пользовательских настроек (`user_preferences`) полем локали с нормализацией/валидацией.
- [ ] Обеспечить backward-compatible сохранение/восстановление runtime snapshot для существующих сессий.
- [ ] Реализовать приоритет резолва локали: явный выбор пользователя -> язык Telegram-профиля -> дефолтная локаль.
- [ ] Добавить пользовательский сценарий переключения языка (в сервисных настройках/команде) без изменения существующих callback-протоколов.

#### Почему этап не ломает общую работоспособность проекта [M]

- При отсутствии новой настройки поведение остается прежним за счет дефолтной локали и безопасного fallback.

#### Проверки/валидация этапа [M]

1. Юнит-тесты нормализации/резолва локали и миграции runtime snapshot.
2. Ручная проверка: смена языка сохраняется и восстанавливается после перезапуска.

#### Ожидаемые артефакты [O]

- Изменения в `core/runtime.py`, `core/runtime_snapshot.py`, `core/user_preferences.py` и соответствующих handler flow.

### Этап 4 - Миграция UI-текстов на ключи локализации

#### Цель этапа [M]

- Перевести пользовательские тексты бота с hardcoded-строк на централизованные ключи локализации.

#### Подзадачи [M]

- [ ] Мигрировать shared UI-копирайт и базовые компоненты (`core/ui_copy.py`, `core/ui_kit/*`, `core/interaction.py`, `core/download_filters.py`).
- [ ] Мигрировать основные handler-потоки (`handlers/common*.py`, `handlers/presets*.py`, `handlers/download*.py`, `handlers/prompt_editor*.py`).
- [ ] Заменить дубли текстов на переиспользуемые ключи и шаблоны с параметрами (DRY/KISS).
- [ ] Сохранить логику сценариев, callback data и FSM-состояния без функциональных изменений.

#### Почему этап не ломает общую работоспособность проекта [M]

- Миграция ограничена presentation-слоем; бизнес-операции генерации/скачивания остаются неизменными.

#### Проверки/валидация этапа [M]

1. Прогон тестов (`pytest`) и проверок качества (`ruff`, `mypy`) после миграции.
2. Ручной smoke-тест ключевых сценариев: `/start`, генерация, пресеты, скачивание, сервисное меню.
3. Проверка отсутствия сырых ключей в UI и корректности fallback при пропущенных переводах.

#### Ожидаемые артефакты [O]

- Обновленные `core/*` и `handlers/*` модули с использованием localization service.

### Этап 5 - Бандлы RU/EN, контроль качества и документация расширения

#### Цель этапа [M]

- Сформировать полноценные RU/EN ресурсные бандлы и закрепить процесс добавления новых локалей без правки ядра.

#### Подзадачи [M]

- [ ] Создать начальные файлы локалей `ru` и `en`: metadata + messages JSON.
- [ ] Добавить тест/валидатор консистентности ключей между `ru` и `en`.
- [ ] Добавить регрессионные тесты на dynamic discovery новых локалей без изменений core-кода.
- [ ] Обновить документацию по структуре локалей и шагам добавления нового языка.

#### Почему этап не ломает общую работоспособность проекта [M]

- Изменения затрагивают ресурсные файлы, тесты и документацию; runtime защищен уже внедренным fallback-механизмом.

#### Проверки/валидация этапа [M]

1. Полный прогон quality gate: `ruff check`, `mypy`, `pytest`.
2. Ручная проверка: добавление тестовой локали только файлами (metadata + JSON) и успешное обнаружение системой.

#### Ожидаемые артефакты [O]

- `locales/ru/*`, `locales/en/*`, тесты локализации, обновленная документация.

### Этап 6 - Полный перевод проекта и устранение hardcoded текстов

#### Цель этапа [M]

- Довести локализацию до полного покрытия проекта: перевести каждый hardcoded пользовательский текст на ключи локализации и исключить зависимость UI от строк в коде.

#### Подзадачи [M]

- [ ] Провести автоматизированную инвентаризацию оставшихся hardcoded строк во всех целевых модулях (`core/*`, `handlers/*`, `application/*`, `infrastructure/*`, `bot.py`, `app_context.py`).
- [ ] Мигрировать все найденные пользовательские строки (сообщения, кнопки, заголовки панелей, alerts, ошибки сценариев) на ключи локализации.
- [ ] Закрыть оставшиеся участки в `prompt_editor*`, `download*`, `presets*`, `common*` и вспомогательных UI-модулях, чтобы не осталось прямых строк в runtime UI-цепочках.
- [ ] Добавить автоматическую проверку/тест-гард на отсутствие новых hardcoded пользовательских строк в целевых runtime-модулях (исключая файлы локалей и тестовые фикстуры).
- [ ] Расширить RU/EN бандлы на все добавленные ключи и подтвердить паритет ключей после полной миграции.

#### Почему этап не ломает общую работоспособность проекта [M]

- Миграция выполняется поверх уже внедренного `LocalizationService` с fallback-механизмом; даже при частичном покрытии ключей runtime остается работоспособным.

#### Проверки/валидация этапа [M]

1. Автоматическая проверка отсутствия hardcoded пользовательских строк в целевых runtime-модулях.
2. Полный прогон quality gate: `ruff check`, `mypy`, `pytest`.
3. Ручной e2e smoke-тест ключевых сценариев: `/start`, `/generate`, `/presets`, `/download`, `/settings`, `/training`, `/cancel` для RU и EN.

#### Ожидаемые артефакты [O]

- Полностью мигрированные модули проекта, расширенные RU/EN ресурсы, тест-гард на hardcoded строки, обновленные отчеты и документация.
