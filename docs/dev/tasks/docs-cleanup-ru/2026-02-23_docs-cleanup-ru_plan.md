## Контекст задачи

### Запрос и результат

- Кратко о запросе: рассортировать документацию проекта, удалить устаревшие материалы, проверить актуальные документы на соответствие текущим возможностям проекта.
- Целевой результат: в репозитории остается только актуальная документация на русском языке, написанная в настоящем времени и без формулировок в стиле "было/стало".

### Границы области работ

- В области работ:
  - Аудит Markdown-документации в `docs/` и корневых документационных файлов (`README.md`, `CONTRIBUTING.md`, `SECURITY.md` и другие релевантные `.md`).
  - Классификация документов: оставить, обновить, удалить.
  - Проверка соответствия документации фактическим возможностям проекта (команды, конфигурация, структура модулей, доступные сценарии).
  - Редактирование текстов в русском языке и в настоящем времени.
  - Удаление устаревших документов и обновление навигации/перекрестных ссылок.
- Вне области работ:
  - Изменение бизнес-логики приложения.
  - Внедрение новых функций для "подгонки" под документацию.

### Основные цели и ограничения

- Файлы/компоненты: `docs/**/*.md`, корневые `*.md`, а также документационные артефакты в `docs/dev/tasks/` только в части этой задачи.
- Затрагиваемые области: документация, процессные артефакты задачи.
- Ограничения и допущения:
  - Содержимое документации приводится к текущему фактическому состоянию кода, а не наоборот.
  - Удаляются только документы, подтвержденные как устаревшие или дублирующие.
  - Технические команды, имена файлов и идентификаторы остаются в оригинальном написании при необходимости точности.

- Лингвистическая унификация документации (русский язык, настоящее время, без ретроспективных формулировок).

## Поэтапный план

### Этап 1 - Инвентаризация и классификация документации

#### Цель этапа

- Получить полный реестр документации и назначить каждому документу статус: оставить, обновить, удалить.

#### Подзадачи

- [x] Собрать полный список Markdown-файлов, относящихся к пользовательской и разработческой документации.
- [x] Сгруппировать документы по назначению (запуск, архитектура, конфигурация, процесс разработки, миграции, отчеты).
- [x] Для каждого документа зафиксировать первичную оценку актуальности и предполагаемое действие.
- [x] Сохранить реестр в артефакт этапа.

#### Почему этап не нарушает работоспособность проекта

- Этап аналитический и не меняет исполняемый код или конфигурацию приложения.

#### Проверки этапа

1. Проверить полноту списка документации по шаблону `*.md` в целевых директориях.
2. Проверить, что у каждого документа есть категория и назначенное действие.

#### Ожидаемые артефакты

- `docs/dev/tasks/docs-cleanup-ru/2026-02-23_docs-cleanup-ru_inventory.md`

### Этап 2 - Сверка документации с возможностями проекта

#### Цель этапа

- Подтвердить фактическую корректность актуализируемых документов относительно текущего состояния проекта.

#### Подзадачи

- [x] Проверить команды, переменные окружения, конфигурационные параметры и архитектурные утверждения документации по коду и структуре репозитория.
- [x] Зафиксировать расхождения и определить формат исправления (переписать, сократить, удалить раздел).
- [x] Обновить план в разделе незапланированных задач при появлении дополнительных действий в рамках области работ.

#### Почему этап не нарушает работоспособность проекта

- Этап верифицирует документацию и готовит безопасные правки без изменения runtime-поведения.

#### Проверки этапа

1. Для каждого документа со статусом "обновить" есть список подтвержденных источников в репозитории.
2. Для каждого расхождения определено конкретное действие по устранению.

#### Ожидаемые артефакты

- Обновленный реестр с пометками сверки в `docs/dev/tasks/docs-cleanup-ru/2026-02-23_docs-cleanup-ru_inventory.md`

### Этап 3 - Обновление и русификация актуальной документации

#### Цель этапа

- Привести все сохраняемые документы к актуальному содержанию на русском языке в настоящем времени.

#### Подзадачи

- [x] Отредактировать документы со статусом "обновить" с учетом подтвержденных расхождений.
- [x] Удалить или переформулировать ретроспективные обороты, включая формулировки в стиле "было/стало".
- [x] Унифицировать стиль изложения: настоящее время, инструкции в явной форме, фактические утверждения.
- [x] Обновить перекрестные ссылки между документами после правок.

#### Почему этап не нарушает работоспособность проекта

- Изменяются только Markdown-файлы документации, рабочая логика приложения не затрагивается.

#### Проверки этапа

1. Выполнить поиск по ключевым ретроспективным формулировкам (`было`, `стало`, `раньше`, `теперь`) и устранить неподходящие случаи.
2. Проверить, что измененные документы сохраняют корректные ссылки на существующие файлы/разделы.

#### Ожидаемые артефакты

- Обновленные документы в `docs/` и корневых `*.md` по результатам инвентаризации.

### Этап 4 - Удаление устаревших документов и финальная валидация

#### Цель этапа

- Удалить подтвержденно устаревшие документы и завершить итоговую проверку консистентности документации.

#### Подзадачи

- [x] Удалить документы, помеченные в реестре как устаревшие/дублирующие.
- [x] Обновить оглавления и навигационные документы после удаления.
- [x] Провести финальную проверку целостности ссылок и единства языковых требований.
- [x] Подготовить итоговый stage report по этапу.

#### Почему этап не нарушает работоспособность проекта

- Удаляются только документационные файлы, признанные неактуальными; исполняемые модули не меняются.

#### Проверки этапа

1. Проверить отсутствие ссылок на удаленные файлы в оставшейся документации.
2. Проверить, что документация, оставшаяся в проекте, соответствует языковым требованиям запроса.

#### Ожидаемые артефакты

- Обновленный набор документации без устаревших файлов.
- Итоговый отчет этапа в `docs/dev/tasks/docs-cleanup-ru/reports/`.

## Риски

| Риск | Влияние | Снижение риска | Ответственный |
| --- | --- | --- | --- |
| Ошибочное удаление полезного документа | Высокое: потеря важной информации для пользователей/разработчиков | Удалять только после фиксации причины в реестре и проверки входящих ссылок | OpenCode |
| Неточная интерпретация текущих возможностей проекта | Среднее: документация расходится с кодом | Сверять утверждения с исходным кодом, конфигами и фактическими командами | OpenCode |
| Неполная языковая унификация | Среднее: смешанный стиль ухудшает читаемость | Применить финальный текстовый аудит по ключевым формулировкам и ручную вычитку | OpenCode |
